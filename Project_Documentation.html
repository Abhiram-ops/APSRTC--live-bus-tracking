PROJECT REPORT ON: APSRTC LIVE BUS TRACKING & MANAGEMENT SYSTEM

ABSTRACT

The public transportation system, particularly the Andhra Pradesh State Road Transport Corporation (APSRTC) in Visakhapatnam, serves as the primary mode of transit for millions of commuters daily. However, the lack of real-time visibility into bus locations leads to unpredictability, longer wait times, and commuter dissatisfaction. This project introduces a comprehensive, real-time "APSRTC Live Tracking System" designed to bring transparency and efficiency to city bus services.

Developed using an advanced technological stack comprising Python (Flask) for the backend RESTful APIs, SQLite for lightweight robust data management, and Vanilla JS paired with Leaflet.js for interactive mapping, the system bridges the gap between the transit authority and the passengers. The application features extensive glassmorphism UI design, offering a premium and responsive user experience. The integrated driver portal allows bus operators to broadcast their GPS location seamlessly using mobile devices, whilst commuters can track active services, view detailed timetables, and plan their journeys securely. Security is enforced through Flask-Limiter, Werkzeug hashing, and secure HTTP headers, making this platform not only attractive but resilient for real-world deployment.

1. INTRODUCTION

1.1 Problem Statement

In rapidly growing urban environments like Visakhapatnam, public transit remains heavily utilized but largely untracked by modern digital standards. Commuters frequently face issues such as uncertain wait times at bus stops, lack of information regarding route changes or delays, and generalized inefficiency in planning their daily travel. Drivers, on the other hand, lack a simple interface to report their active status without the need for expensive dedicated hardware installed in the buses.

1.2 Proposed Solution

This project proposes a dual-interface web application. First, a Commuter Portal where users can digitally query routes between dynamic stations, view real-time location markers of matching buses, and study route timetables. Second, a Driver Dashboard accessible via standard smartphones, utilizing the HTML5 Geolocation API to transmit coordinates to a central database at regular intervals. By harnessing cloud-deployment paradigms on platforms like Render, the system operates constantly with near-zero downtime.

1.3 Core Objectives

  • To develop a real-time tracking interface showing bus movements on interactive digital maps.
  • To establish a highly responsive and visually appealing user interface utilizing modern design tokens (Glassmorphism).
  • To implement a stateless but highly concurrent mechanism to calculate and update bus locations globally.
  • To provide robust authentication systems for secure Driver and Administrator logins.

2. SYSTEM ARCHITECTURE & TECHNOLOGY STACK

2.1 Hardware and Software Requirements

  • Server Environment: Linux/Windows based Cloud Server (Render / Azure).
  • Backend Engine: Python 3.10+ coupled with the Flask WSGI Microframework.
  • Database Management: SQLite3 facilitated via Flask-SQLAlchemy (ORM).
  • Frontend Technologies: HTML5, CSS3, Vanilla ECMAScript 6 (JavaScript).
  • Mapping Engine: OpenStreetMap data rendered via Leaflet.js.

2.2 Architecture Design

The system follows the traditional Client-Server Architecture paired with the Model-View-Controller (MVC) paradigm. The browser acts as the View layer rendering dynamic Jinja2 templates. The Flask application instance serves as the Controller, orchestrating inputs from users (route searches, login attempts) and interacting with the SQLAlchemy Models. Gunicorn acts as the production-grade HTTP server managing worker threads to sustain high volumes of concurrent requests without lagging.

3. DETAILED MODULE DESCRIPTION

3.1 The Commuter Module

This represents the primary public-facing interface. It features a scenic Vizag-themed graphical wrapper optimized for fast loading over cellular networks. Commuters input their "From" and "To" destinations. The server evaluates route intersections, fetching all applicable "Services" that connect those stops. Users can then toggle a live map view which parses GeoJSON data and plots buses moving in real-time. Estimated Times of Arrival (ETA) and scheduled timetables are also displayed contextually based on the user's specific query.

3.2 The Driver Module

Constructed with a 'Mobile-First' philosophy, this module features large interactive touch targets. Drivers securely log in utilizing credentials verified against hashed strings in the database. Upon starting a trip, a background JavaScript daemon invokes the device's native GPS unit, streaming coordinates to a dedicated API endpoint asynchronously. The visual indicator transitions to an "Active" state confirming successful broadcast.

3.3 Security & Authentication Module

Security is paramount due to the collection of location data. This module integrates:

  • Password Cryptography: Utilizing Werkzeug security libraries for salted hash storage.
  • Rate Limiting: Enforced by Flask-Limiter to throttle repeated failed authentication requests, preventing brute-force intrusion.
  • Session Integrity: Hardened server-side session cookies that manage state securely.
  • Prevention of Injection: Overarching reliance on SQLAlchemy paradigms ensuring that all inputs are parameterized before SQL execution.

4. DATABASE SCHEMA DESIGN

The relational database relies on tightly normalized tables representing the transport network logic:

Entity Name Primary Function and Relationships
Routes Maintains core paths (`route_id`, `from_station`, `to_station`). It dictates the overarching journey path independently of individual buses.
Services Represents scheduled trips under a Route. Holds price, service type (e.g., Express, Ordinary), and links rigidly to Timetables.
Vehicles & Location The `Vehicles` table identifies actual physical buses (`vehicle_no`, `status`), whilst the `LiveLocation` table handles volatile, rapidly updating coordinate metrics (`lat`, `lng`, `speed`, `updated_at`).
Drivers & Users Maintains encrypted identities. Drivers are relationship-mapped specifically to `Services` to authorize location publishing for that exact trip.

5. IMPLEMENTATION & DEPLOYMENT

The deployment pipeline involves version control through GitHub and Continuous Deployment (CD). Utilizing `gunicorn` with optimized worker threads bridges the WSGI application with the external network. Environment variables (`.env`) shield sensitive data such as secret keys. The repository uses `requirements.txt` to strictly manage dependencies like Flask-SQLAlchemy and python-dotenv to prevent runtime discrepancies across environments.

6. CONCLUSION

The APSRTC Live Tracking system successfully demonstrates the feasibility of combining lightweight, open-source mapping services with robust Python backend technologies. By delivering a highly responsive, glassmorphism-themed frontend alongside a secure, efficient data pipeline, the project modernizes the transit experience for commuters in Vizag. It provides an impressive foundation that achieves the goal of real-time visibility without requiring significant hardware investment by the authorities.

Future Enhancements

Future scope for this application includes implementing Machine Learning algorithms to predict bus arrival times accurately despite dynamic traffic congestion, incorporating real-time traffic layer overlays on Leaflet.js, and integrating payment gateways for digital ticket acquisition.
